# Ratziel 用户文档更新计划

## 1. 更新前提

### 1.1 文档定位
本轮更新后的文档**主要面向使用者**，包括：

- 服主
- 配置作者
- 资源包/玩法作者
- 希望快速上手 Ratziel 的普通用户

本轮文档**不以开发者扩展为主线**，因此会遵循以下原则：

1. 优先回答“怎么用”而不是“怎么实现”
2. 优先给出可复制示例，而不是展开底层源码细节
3. 只保留用户理解功能所必需的概念说明
4. 将开发者视角内容降级为附录、注释或后续专题

### 1.2 命名规则修订
文档中的元素命名规则统一调整为：

- **允许跨元素类型使用相同的元素名**
- **不允许同一元素类型下出现多个同名元素**

文档中所有涉及“元素重名”“命名冲突”“组织建议”的表述，都要按这条规则重写。

### 1.3 更新目标
本轮更新不是简单补充页面，而是要把现有文档从“零散条目说明”整理为一条清晰的用户学习路径：

- 安装 → 工作空间 → 创建第一个物品 → 数据/标签/动作 → 模板 → 脚本 → 高级功能 → 速查

---

## 2. 当前文档主要问题

根据现有 Wiki 与代码现状，当前文档存在以下问题：

### 2.1 用户入口不够清晰
当前已有以下页面：

- `docs/intro.md`
- `docs/start/*`
- `docs/advance/*`
- `docs/script/*`
- `docs/quick-lookup/*`

但入口页内容过少，缺少一条完整的学习路径，用户看完后仍然不知道：

- 配置应该放哪里
- 一个文件最小能怎么写
- 先学什么，后学什么
- 物品是怎么一步步配置出来的

### 2.2 部分内容已经过时
现有文档中有多处字段名或功能描述与当前代码不一致，例如：

- 工作空间配置字段仍使用旧表达，如 `use-filename`、`unified-type`
- 脚本支持页仍写着 `KotlinScripting (暂不支持)`
- 一些页面对元素概念的定位偏旧，不符合当前实现

### 2.3 术语切入仍偏开发者视角
例如“元素”“组件”“解释器”“物品源”等术语，在现有文档中有时直接出现，但没有充分转换为用户可以理解的任务式表达。

### 2.4 页面之间的衔接不足
目前页面多为单点说明，缺少：

- 从入门到进阶的顺序设计
- 页面间前置/后续关系提示
- 示例在多个页面中的一致复用

### 2.5 命名规则需要统一修正
现有文档中关于元素命名冲突的说明需要统一改写为：

- 同类型唯一
- 跨类型可重名

## 3. 本轮更新的总体策略

### 3.1 结构策略
采用“三层结构”：

1. **入门主线**：用户第一次使用时必须阅读的页面
2. **功能专题**：模板、动态物品、脚本等进阶能力
3. **速查附录**：字段、触发器、物品源、标签解析器等表格式参考

### 3.2 表达策略
统一采用以下写法：

- 先给最小示例
- 再解释节点含义
- 然后给典型用法
- 最后补充注意事项/坑点

避免一上来直接讲源码、注册机制、生命周期内部实现。

### 3.3 示例策略
示例统一分为三类：

1. **最小示例**：能直接抄
2. **组合示例**：展示功能协作
3. **完整案例**：可作为模板继续扩写

### 3.4 一致性策略
本轮更新要求所有页面统一：

- 统一字段名
- 统一命名规则
- 统一脚本语言列表
- 统一“工作空间/元素/物品”的关系表达
- 统一示例命名风格

## 4. 页面结构调整方案

## 4.1 保留并重写的主栏目

### A. 入门指南 `docs/start/*`
作为用户主线阅读区。

建议目标顺序：

1. 配置插件
2. 工作空间与元素文件
3. 创建第一个物品
4. 物品动作
5. 物品数据
6. 物品标签

### B. 进阶功能 `docs/advance/*`
保留给用户真的会用到的高级能力。

目前建议：

- 模板继承
- 动态物品
- 后续可补：静态物品 / 多层物品 / 外部物品源进阶使用

### C. 脚本系统 `docs/script/*`
保留为“用户如何使用脚本”的专题，不展开成底层实现文档。

### D. 文档速查 `docs/quick-lookup/*`
用于字段、触发器、物品源、组件、解析器的快速查询。

## 4.2 作为草稿素材使用但不作为主入口的栏目

### `docs/writing/*`
此目录更适合作为：

- 编写中的草稿
- 长篇专题的素材源
- 尚未整理完的内容暂存区

本轮不把它作为最终用户主入口，而是：

1. 继续保留
2. 从中抽取成熟内容并迁移到正式栏目
3. 不让用户在导航上优先进入该栏目
