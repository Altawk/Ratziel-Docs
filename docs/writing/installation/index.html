安装与部署

这篇文档只回答一件事：

怎样把 Ratziel 真正装到服务器里，并跑起来。

如果你还没开始写配置，建议先看这页，再去看：

  • 快速开始（重写版）
  • 配置与工作空间（重写版）

先理解你手里的是什么​

Ratziel 当前公开仓库更像是：

  • 一套源码
  • 一套文档
  • 一套可自行构建的插件工程

也就是说，你的安装方式通常分两种：

情况 A：你已经拿到了打包好的 jar​

这种情况最简单，直接放进服务器的 plugins 目录即可。

情况 B：你拿到的是源码仓库​

这种情况需要你先本地构建，再把构建产物放进服务器。

这篇文档会把两种情况都覆盖到。

基础环境要求​

根据当前工程配置，可以明确确认这些要求：

  • 平台：Bukkit / Paper 生态
  • Java 目标版本：Java 8
  • 推荐服务端：优先使用较新的 Paper

文档默认按 Bukkit/Paper 服务器来写。

可选兼容插件​

当前 Bukkit 平台声明了这些可选依赖：

  • PlaceholderAPI
  • CraftEngine
  • NeigeItems

它们不是安装 Ratziel 的硬前置，但如果你要用对应兼容功能，就需要自己安装。

是否需要单独安装 TabooLib​

通常不需要。

当前 Bukkit 平台构建里已经把 TabooLib 运行环境打进插件产物流程里，正常使用时不需要再额外把 TabooLib 当独立前置插件安装。

方式一：直接安装已打包的 jar​

如果你已经拿到构建好的插件文件：

  1. 关闭服务器
  2. 把 jar 放进：
plugins/
  1. 启动服务器
  2. 等待插件首次生成目录和默认文件

启动成功后，你通常会在插件目录下看到类似结构：

plugins/Ratziel/

然后就可以继续看：

  • 快速开始（重写版）

方式二：从源码自行构建​

如果你拿到的是源码仓库，按 README 的方式构建即可。

Windows​

gradlew.bat clean build

macOS / Linux​

./gradlew clean build

构建结果在哪里​

当前 Bukkit 平台构建脚本会把打包结果放到根目录下的：

build/

文件名格式类似：

Ratziel-Bukkit-版本号.jar

构建完成后，把这个 jar 放进服务器的 plugins 目录即可。

第一次启动后要做什么​

第一次启动完成后，建议按这个顺序检查：

1. 看插件目录有没有生成​

确认是否出现：

plugins/Ratziel/

2. 看配置文件是否生成​

最重要的是：

  • settings.yml
  • 工作空间目录

3. 先不要急着写很复杂的配置​

先用最小示例确认基础流程跑通。

直接继续看：

  • 快速开始（重写版）

安装后的推荐检查顺序​

第一步：确认命令可用​

进入服务器后先尝试：

/ratziel reload

如果主命令可用，说明插件至少已经正常加载到命令层。

第二步：确认工作空间可读​

在工作空间中放一个最小物品示例，然后重载。

第三步：确认物品能正常生成​

/r-item give BlackApple

如果能成功拿到物品，说明安装和基础读取流程都通了。

常见安装误区​

1. 把源码仓库直接丢进 plugins​

不行。

源码仓库要先构建成 jar，才能给服务器加载。

2. 还沿用旧文档里的 config.yml / items/​

当前新版文档主线应以：

  • settings.yml
  • workspace

为准，不再按旧的 items/ 目录思路理解。

3. 以为 PlaceholderAPI 是硬前置​

不是。

只有当你要使用 {papi:...} 或相关兼容能力时，才需要额外安装它。

4. 启动成功就等于配置也一定正确​

不一定。

插件能启动，只说明主体能加载；配置是否被正确读取，还需要后续用 reload、element list、r-item give 去验证。

5. 把“安装完成”和“能做出物品”混为一谈​

真正算跑通，至少要完成：

  1. 插件加载成功
  2. 配置能被读取
  3. 物品能被生成
